Actividad 2 - Módulo 4

1. ¿Qué son los Modelos de Estimación?

Los Modelos de estimación son métodos utilizados para prever el esfuerzo, tiempo y costo necesario para completar un proyecto. Estos modelos se basan en diversos factores, como el tamaño del proyecto, la complejidad y el historial de proyectos anteriores, para ofrecer estimaciones más precisas y ayudar en la planificación y gestión del proyecto.

2. Investiga sobre diferentes modelos de estimación utilizados en la industria del desarrollo de software

Modelo COCOMO

COCOMO (Constructive Cost Model) es un modelo de estimación desarrollado por Barry Boehm. Se basa en una fórmula matemática que utiliza el tamaño del software (en líneas de código) para estimar el esfuerzo, el costo y el tiempo de desarrollo. Existen diferentes versiones de COCOMO, como COCOMO I, II y III, que se adaptan a diferentes tipos de proyectos y niveles de detalle.

Método de Puntos de Función

El método de Puntos de Función mide el tamaño del software en términos de las funcionalidades que ofrece al usuario. Se basa en contar el número de entradas, salidas, consultas, archivos y otros componentes del sistema. Los puntos de función permiten estimar el esfuerzo y el costo basándose en la complejidad funcional del software.

Comparación

El modelo COCOMO es más adecuado para estimaciones basadas en el tamaño del código fuente y es útil en proyectos donde se tiene un historial previo de desarrollo. Por otro lado, el método de Puntos de Función es más adecuado para estimar proyectos basados en funcionalidades y no requiere información sobre el tamaño del código, lo que lo hace útil en las primeras etapas del desarrollo.

3. ¿A qué llamamos la estimación de un proyecto de software?

La estimación de un proyecto de software es el proceso de prever los recursos necesarios, como el tiempo y el costo, para desarrollar un software. Incluye la evaluación de los requisitos del proyecto, el análisis de la complejidad y el cálculo del esfuerzo requerido para completar todas las tareas del proyecto.

4. Enumera y explica brevemente los principios esenciales para el desarrollo de sistemas

  • Modularidad: Dividir un sistema en módulos independientes que puedan desarrollarse, probarse y mantenarse por separado.
  • Cohesión: La medida en que las partes de un módulo están relacionadas y trabajan juntas para cumplir una única tarea.
  • Acoplamiento: El grado de dependencia entre módulos. Se busca un bajo acoplamiento para facilitar el mantenimiento y la evolución del sistema.
  • Abstracción: Ocultar los detalles de implementación y mostrar solo la funcionalidad esencial a los usuarios o desarrolladores.

Ejemplo de Cohesión

En el desarrollo de un sistema de gestión de inventario, un módulo de gestión de productos podría tener alta cohesión si todas sus funciones (añadir productos, eliminar productos, actualizar información de productos) están relacionadas con la gestión de productos y se realizan dentro del mismo módulo. Esto facilita el mantenimiento y la comprensión del código.

5. Lista de partes interesadas en un proyecto de desarrollo de software para una aplicación móvil

  • Clientes: Definen los requisitos y expectativas del producto. Esperan una aplicación funcional y eficiente que satisfaga sus necesidades.
  • Usuarios Finales: Interactúan directamente con la aplicación. Buscan una interfaz intuitiva y características que faciliten su experiencia.
  • Desarrolladores: Encargados de la implementación técnica de la aplicación. Necesitan especificaciones claras y un entorno de trabajo adecuado.
  • Diseñadores de UI/UX: Se ocupan del diseño de la interfaz y la experiencia del usuario. Buscan comprender los requisitos del usuario para crear un diseño atractivo y funcional.
  • Equipo de Control de Calidad: Prueba la aplicación para asegurar que funcione correctamente y cumpla con los requisitos. Buscan informes detallados y un entorno para realizar pruebas exhaustivas.
  • Gerentes de Proyecto: Supervisan el progreso del proyecto y aseguran que se mantenga dentro del presupuesto y el cronograma. Necesitan informes regulares y una planificación adecuada.
  • Soporte Técnico: Proporciona asistencia a los usuarios y resuelve problemas técnicos. Necesitan documentación completa y acceso a recursos para la resolución de problemas.

6. Diseña un plan de proyecto en el que se definan las diferentes fases y actividades necesarias para desarrollar un sistema de gestión de inventario

Fases y Actividades

  • Fase de Planificación:
    • Definición de requisitos: Reunir y documentar los requisitos del sistema.
    • Desarrollo del plan del proyecto: Crear un cronograma y asignar recursos.
  • Fase de Diseño:
    • Diseño de arquitectura del sistema: Definir la estructura general del sistema.
    • Diseño de la interfaz de usuario: Crear prototipos y diseños de la interfaz.
  • Fase de Desarrollo:
    • Implementación de módulos: Programar las diferentes partes del sistema.
    • Integración del sistema: Unir los módulos y realizar pruebas iniciales.
  • Fase de Pruebas:
    • Pruebas unitarias: Probar cada módulo individualmente.
    • Pruebas de integración: Verificar que los módulos funcionen correctamente juntos.
    • Pruebas de aceptación: Asegurarse de que el sistema cumple con los requisitos del cliente.
  • Fase de Despliegue:
    • Preparación para el lanzamiento: Configurar el entorno de producción y realizar la capacitación necesaria.
    • Despliegue del sistema: Lanzar el sistema para su uso en producción.
  • Fase de Mantenimiento:
    • Monitoreo y soporte: Supervisar el sistema y proporcionar soporte técnico.
    • Actualizaciones y mejoras: Realizar actualizaciones y mejoras según sea necesario.

7. Elabora un conjunto de normas y estándares que deben seguirse en el desarrollo de software para garantizar la consistencia en la codificación y la documentación

  • Normas de Codificación: Utilizar convenciones de nombres claras y consistentes. Por ejemplo, usar el estilo de nombres camelCase para variables en JavaScript.
  • Documentación: Documentar el código y los procesos de desarrollo. Por ejemplo, usar comentarios en el código para explicar la funcionalidad y generar documentación técnica utilizando herramientas como Javadoc.
  • Control de Versiones: Usar sistemas de control de versiones como Git para gestionar cambios en el código. Por ejemplo, seguir una estrategia de ramificación para facilitar el desarrollo paralelo y la integración de cambios.
